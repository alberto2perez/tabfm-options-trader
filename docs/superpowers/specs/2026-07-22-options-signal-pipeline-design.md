# Options Signal Pipeline — Design Spec
**Date:** 2026-07-22
**Status:** Approved

---

## Overview

A nightly Python pipeline that uses TabFM's in-context learning (ICL) to scan a watchlist of 20–30 tickers, score every candidate vertical spread, and emit a single options trade recommendation with a baked-in probability of profit (POP%). All recommendations are executed in a local paper trading journal before risking real capital. A backtesting module validates the signal against historical data before live paper trading begins.

**Capital constraint:** $1,000 maximum loss per trade, enforced at the filter layer.

---

## Architecture

The system has eight sequential stages plus a pluggable Data Adapter layer that switches between live (Robinhood MCP) and historical (yfinance + Black-Scholes) data sources. This single abstraction makes the backtest identical to the live pipeline — same code, different adapter.

```
          ┌─────────────────────────────────────┐
          │         Data Adapter Layer          │
          │  LiveAdapter        HistAdapter     │
          │  (Robinhood MCP)    (yfinance +     │
          │                      Black-Scholes) │
          └──────────────┬──────────────────────┘
                         │  same interface
                         ▼
[1] Chain Fetcher
[2] Feature Engineer
[3] Context Builder
[4] TabFM Scorer
[5] Trade Recommender
[6] Paper Executor
[7] Position Auditor
[8] Accuracy Tracker
                         ▲
              ┌──────────┴────────────┐
              │   Backtest Runner     │
              │  iterates over dates  │
              │  T-252 → T (1 yr)     │
              │  enforces no-lookahead│
              └───────────────────────┘
```

### Stage Responsibilities

**[1] Chain Fetcher** — pulls the live options chain and underlying closing price for each watchlist ticker via the Data Adapter. Outputs raw chain data per ticker.

**[2] Feature Engineer** — computes the full feature row (see Feature Schema) for each candidate vertical spread on each ticker. Candidate spreads are generated by iterating over strike pairs within the delta range 0.15–0.40 for the short leg, at both weekly and monthly expiries.

**[3] Context Builder** — selects the 60 most regime-similar historical rows from the local Parquet store as the TabFM context window. Regime similarity is defined by exact matching on `vix_bucket` + `trend_direction` + `iv_regime`. If fewer than 60 regime-exact rows exist, the builder relaxes one dimension at a time (drops `iv_regime` first, then `trend_direction`) until 60 rows are found or all available history is used. The historical store is bootstrapped using yfinance during the first 60 calendar days of operation.

**[4] TabFM Scorer** — runs two TabFM models in parallel:
- `TabFMClassifier` → predicts `profitable` (POP%)
- `TabFMRegressor` → predicts `return_pct` (expected return on risk)

**[5] Trade Recommender** — applies the filter gauntlet, scores survivors, and emits exactly one trade recommendation per night.

**[6] Paper Executor** — logs the recommendation to the local SQLite journal as a simulated entry at mid-price. Prints the formatted recommendation to stdout.

**[7] Position Auditor** — runs nightly alongside [1]–[6]. For each open paper trade, checks if expiry has passed and resolves the outcome by comparing underlying price to strikes. Status values: `won` (short strike expired OTM, full credit kept), `lost` (short strike expired ITM, max loss), `partial` (trade closed early at 50% of max profit — the standard premium-selling management rule). Writes `actual_pnl` and `date_closed` back to SQLite.

**[8] Accuracy Tracker** — queryable at any time. Reads the SQLite journal and computes win rate, POP calibration, cumulative P&L, per-regime breakdown, and drawdown.

## Data Adapter Interface

Both adapters expose the same interface:

```python
class DataAdapter:
    def get_underlying(self, ticker: str, as_of: date) -> dict
    def get_options_chain(self, ticker: str, as_of: date) -> pd.DataFrame
    def get_vix(self, as_of: date) -> float
```

`LiveAdapter` calls Robinhood MCP. `HistAdapter` calls yfinance for prices and computes approximate IV from VIX scaled to ticker beta. Black-Scholes prices synthetic option chains from this IV.

---

## Backtest Runner

Iterates over trading dates from `T-252` to `T-1` (one year of history). For each date:
1. Instantiates `HistAdapter(as_of=date)`
2. Runs stages [1]–[6] with a hard no-lookahead boundary: the Context Builder may only access rows where `date < as_of`
3. The Position Auditor resolves outcomes using `HistAdapter(as_of=expiry_date)`

The no-lookahead rule is enforced at the adapter level with an assertion, not assumed by convention.

**Historical options data:** The `HistAdapter` uses Black-Scholes reconstruction with yfinance OHLCV + VIX as the IV proxy. This is an approximation sufficient for signal validation. Once backtesting shows edge, Polygon.io (~$30/month) provides full chain history for higher-fidelity validation.

## Feature Schema

One row per candidate vertical spread per nightly run.

### Underlying Features (numerical)

| Column | Description |
|---|---|
| `price_close` | Closing price of underlying |
| `momentum_5d` | 5-day return % |
| `momentum_20d` | 20-day return % |
| `atr_14` | 14-day Average True Range |
| `volume_zscore` | Today's volume vs 20-day average (standard deviations) |
| `price_vs_sma20` | % above/below 20-day moving average |
| `vix_level` | VIX index value at close |
| `vix_5d_change` | VIX 5-day delta |

### Volatility Features (numerical)

| Column | Description |
|---|---|
| `iv_rank` | IV percentile over past 252 days (0–100) |
| `hv20` | 20-day realized historical volatility |
| `hv_iv_ratio` | HV20 / IV — below 1.0 means options are expensive vs realized vol |

### Regime Features (categorical)

| Column | Values |
|---|---|
| `vix_bucket` | `low` (<15) / `normal` (15–25) / `elevated` (25–35) / `spike` (>35) |
| `trend_direction` | `uptrend` / `downtrend` / `sideways` |
| `iv_regime` | `cheap` (<25th pct) / `fair` / `expensive` (>75th pct) |
| `earnings_flag` | `earnings_week` / `no_earnings` |
| `sector` | `index_etf` / `tech` / `finance` / `energy` / `consumer` |

### Spread Structure Features (mixed)

| Column | Type | Description |
|---|---|---|
| `dte` | numerical | Days to expiry |
| `short_delta` | numerical | Delta of the short strike |
| `strike_distance_pct` | numerical | Short strike distance from spot as % |
| `spread_width_dollars` | numerical | Max loss per contract in dollars |
| `bid_ask_pct` | numerical | Bid/ask spread as % of max profit |
| `direction` | categorical | `call_spread` / `put_spread` |
| `expiry_type` | categorical | `weekly` / `monthly` |

### Target Labels

- `TabFMClassifier` → `profitable`: 1 if short strike expires OTM, 0 if breached
- `TabFMRegressor` → `return_pct`: actual P&L / max risk (captures partial outcomes)

## Filter Gauntlet & Scoring

Applied in [5] Trade Recommender before scoring:

```
Filter 1: spread_width_dollars × contracts ≤ $1,000   (hard cap)
Filter 2: bid_ask_pct ≤ 15%                           (liquidity)
Filter 3: open_interest ≥ 100 at both strikes         (fillability)
Filter 4: DTE between 7 and 45                        (avoid 0DTE and illiquid LEAPS)
Filter 5: short_delta between 0.15 and 0.40           (structural range)
Filter 6: no earnings within DTE window               (skip IV crush uncertainty)

Score = POP% × expected_return_pct
Winner = argmax(Score)
```

Contract count is set to `floor($1,000 / (spread_width × 100))`, capped at 10 to avoid concentration.

## Paper Trading Journal (SQLite)

**File:** `trading/paper_journal.db`

```sql
CREATE TABLE paper_trades (
  trade_id      INTEGER PRIMARY KEY,
  date_entered  DATE,
  ticker        TEXT,
  direction     TEXT,
  strike_short  REAL,
  strike_long   REAL,
  expiry        DATE,
  dte           INTEGER,
  entry_credit  REAL,
  spread_width  REAL,
  contracts     INTEGER,
  max_loss      REAL,
  max_profit    REAL,
  pop_predicted REAL,
  exp_return    REAL,
  regime        TEXT,
  status        TEXT DEFAULT 'open',
  actual_pnl    REAL,
  date_closed   DATE
);
```

Simulated fill is always at mid-price (conservative assumption — real fills may be better or worse).

## Accuracy Tracker Metrics

| Metric | Description |
|---|---|
| Win rate | Actual wins / total closed trades |
| POP calibration | Predicted avg POP% vs actual win rate (ideally within 5%) |
| Cumulative paper P&L | Running sum of `actual_pnl` |
| Max drawdown | Worst consecutive loss streak in dollar terms |
| Best regime | `vix_bucket × trend_direction × iv_regime` combo with highest win rate |
| Worst regime | Combo to avoid — surfaces a "no-trade" rule for the live system |

## Bootstrap Sequence

1. Run backtest over T-252 → T-1 using `HistAdapter` to validate signal has edge
2. If backtest win rate ≥ 55% with POP calibration within 10%, proceed to paper trading
3. Run live pipeline nightly for 30 trading days (≈6 weeks) using `LiveAdapter`
4. Review Accuracy Tracker: if paper win rate ≥ 55% and max drawdown < $500, consider transitioning to real capital with 1-contract sizing

## Out of Scope

- Real money execution (Robinhood order placement)
- Real-time intraday scanning
- Multi-leg strategies beyond vertical spreads (iron condors, calendars)
- Fine-tuning or retraining TabFM weights
